As described Corbado Observe works by sending events from your application. To fully understand how those events are structured and work you first need to understand the data model. The following diagram provides an overview of the data model:

(DIAGRAM TODO)

Projects:

Projects are the top-level entities in the data model. They contain all data and are the storage for all events. In Corbado you can create multiple projects. They have nothing to do with each other.

Applications:

Applications are entities that represent your application. E.g. if you have a website and an native app you usually do not create two different projects but create multiple applications. Read our section on [Applications](/corbado-observe/tracking/applications) for more details.

Flows:

Flows are composed of decisions and subflows and describe a longer journey of your user like a login flow or recovery flow. Read our section on [Flows](/corbado-observe/tracking/flows) for more details.

Decisions:

Decisions are used to track systen decisions and user decisions. For example if you represent a list of login methods (password, email OTP and social login) you can track this decisions. In the funnel analysis this decision point will be a distinct node which increases visibility and understanding and you can filter the funnel by decisions. Decisions are created by sending special events. Read our section on [Decisions](/corbado-observe/tracking/decisions) for more details.

Subflows:

Subflows are smaller building blocks of flows. They combine events into a unit for e.g. a passkey login or recovery step. Read our section on [Subflows](/corbado-observe/tracking/subflows) for more details.

Events:

From a tracking perspective everything is done by sending events. There are different types of events and different events then create flows, subflows or decisions.
